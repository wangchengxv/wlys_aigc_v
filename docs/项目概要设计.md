# 项目概要设计

## 1. 设计目标

本文档从技术与模块视角说明当前项目如何承接“校园教学 + AIGC 项目生产 + 平台治理”的整体需求，重点描述设计原则、总体架构、模块协作、关键流程、核心数据对象与扩展方向，作为项目级概要设计基线。

## 2. 总体设计原则

### 2.1 平台一体化

系统将创作、教学、项目、治理与模型配置统一到同一前后端应用中，通过首页、导航分组和统一认证上下文组织功能入口，避免拆成多个互相割裂的小系统。

### 2.2 项目中心化

涉及长链路创作的能力均围绕 `ScriptProject` 聚合展开，而不是继续停留在单次生成任务层面。剧本、资产、关键帧、镜头、视频、配音、成片和导出都归属于同一个项目上下文。

### 2.3 教学上下文化

课程、作业、提交、评审不是独立子系统，而是项目生产的重要上层语义。项目既可以作为自由创作单元存在，也可以被课程作业和提交记录引用，用于教学管理与成果验收。

### 2.4 能力路由化

所有 AI 能力按文本、图像、视频、TTS、口型同步等逻辑步骤组织，通过能力路由与项目级覆盖共同决定模型来源，避免业务层直接写死某个供应商。

### 2.5 阶段异步化

视频、配音、口型同步、成片、导出等耗时操作统一采用任务化和异步编排，既支持并发执行，也支持失败重试和阶段状态查询。

### 2.6 审核前置化

项目从生产走向交付前需要经过内容审核和状态确认，保证教学成果或导出产物能够被教师和管理员纳入统一治理。

## 3. 总体架构

### 3.1 分层结构

项目整体采用“前端工作台 + 后端业务服务 + 存储与外部模型网关”三层结构：

1. 前端层：`aigc-site-react`，负责角色化导航、页面编排、表单交互、状态展示与调用后端 API。
2. 后端层：`aigc-server`，负责认证授权、业务规则、项目编排、文件管理、审计与运营统计。
3. 基础设施层：数据库、本地文件存储、外部模型供应商接口、统一路由代理。

### 3.2 前端分区

从路由与导航可识别出以下前端功能分区：

- 概览区：首页、运营看板、登录入口。
- 创作工具区：工作台、无限画布、三视图工具、历史记录。
- 课程与实训区：课程列表、课程详情、作业详情、提交与评分。
- 项目生产区：剧本工程列表、创建、详情、预览、资产、视频、配音、口型同步、成片、导出、流程工作区。
- 资源与治理区：组织用户、媒体资源、审计日志、设置、模型与服务商。

### 3.3 后端领域划分

从控制器与服务命名可抽象为以下领域模块：

- 认证与权限域：`AuthController`、`RequestAuthService`、`AuthorizationService`。
- 创作生成域：`GenerationController`、生成服务、历史记录与模型配置。
- 教学管理域：课程、作业、提交、评审。
- 项目生产域：`ScriptProjectController`、`ScriptAssetController`、`KeyframeController`、`VideoPipelineController`。
- 治理运营域：组织目录、媒体资源、审计日志、运营看板、内容审核。
- 路由与模型域：连接配置、模型配置、服务商目录、路由控制台、兼容代理接口。
- 文件与存储域：文件预览/下载、本地资产管理、项目聚合保存。

## 4. 模块协作设计

### 4.1 认证与权限模块

认证模块负责从请求头中恢复用户上下文，形成 `RequestUserContext`。权限模块基于角色、组织、课程归属与项目拥有关系判断操作是否合法。

协作规则如下：

- 管理员拥有全局治理能力。
- 教师具备课程管理、项目查看、评审和看板访问能力。
- 学生主要拥有个人项目生产、提交和查看自身提交结果的能力。
- 项目读写权限与课程评审权限联动，避免无关用户访问他人项目。

### 4.2 课程管理模块

课程管理模块由课程、作业、提交三个层次组成：

- 课程对象承接教师、组织和课程说明。
- 作业对象承接教学任务和状态要求。
- 提交对象承接学生项目与评分结果。

课程模块与项目模块的关系是“上层组织与验收关系”，即课程并不直接承担项目生产逻辑，而是通过 `courseId`、`assignmentId`、`submissionId` 关联项目成果。

### 4.3 创作工具模块

创作工具模块为用户提供轻量快速生成入口，包括工作台、图像/视频生成、图生视频、无限画布与三视图工具。该模块适合作为：

- 学生进入项目生产前的快速尝试环境。
- 教师辅助教学演示与素材生成的工具箱。
- 平台能力演示与轻量试用入口。

这部分能力与项目域并行存在，遵循“快速生成不必进入项目，长链路生产必须进入项目”的边界。

### 4.4 项目生产模块

项目生产模块是系统的核心编排中心，围绕 `ScriptProjectAggregate` 聚合工作。聚合中已经包含：

- 项目元信息。
- 修订记录与文档版本。
- 资产、关键帧、镜头、视频任务。
- 配音、口型同步、剪辑渲染、成片、导出包任务。
- 内容审核记录与流水线运行记录。

该设计说明系统采用“一个项目承载多阶段产物”的方式，而不是为每一步单独拆出完全独立的项目容器。

### 4.5 路由与模型模块

路由与模型模块负责把不同业务步骤映射到正确的模型能力。其职责包括：

- 维护服务商与模型配置。
- 管理 API Key、路由优先级、日志与统计。
- 根据工作流函数键选择项目级覆盖模型、显式能力模型或平台默认模型。
- 兼容外部 OpenAI/Anthropic 风格调用，统一出口。

这一模块是平台长期扩展能力的关键基础设施。

### 4.6 资源与治理模块

资源与治理模块负责沉淀平台运行结果：

- 媒体资源中心统一查看文件记录。
- 审计日志记录关键治理动作。
- 运营看板汇总课程、提交、项目、审核与交付状态。
- 内容审核在项目进入交付前形成最终把关点。

## 5. 关键流程设计

### 5.1 教学主流程

1. 教师创建课程并发布作业。
2. 学生进入课程并创建或关联项目。
3. 学生在项目中完成剧本与视频生产。
4. 学生提交作业成果。
5. 教师查看提交、评分、给出评审意见。
6. 优秀成果可继续进入审核、成片与导出。

### 5.2 项目生产主流程

1. 创建剧本工程或上传文本文件。
2. 生成原始剧本文件并保存项目元数据。
3. 发起剧本完善、改写、续写或三阶段优化。
4. 形成结构化脚本与修订记录。
5. 抽取人物、背景、道具等资产。
6. 生成关键帧、视觉提示词、三视图或九宫格素材。
7. 拆分镜头并生成视频分段任务。
8. 查询视频任务状态并对失败分段重试。
9. 在视频结果基础上继续生成配音与口型同步。
10. 进入剪辑预览、剪辑发布、最终成片与导出包阶段。

### 5.3 审核与交付流程

1. 项目拥有者提交内容审核。
2. 教师或管理员查看审核状态。
3. 审核通过后允许继续导出；审核驳回则记录意见并等待修改。
4. 导出任务生成交付包，供下载或归档。

### 5.4 模型调用流程

1. 前端发起具体业务动作，如剧本完善或关键帧生成。
2. 后端根据工作流步骤确定能力类型和函数键。
3. 系统优先读取项目级工作流覆盖模型。
4. 若未配置覆盖，则回退到项目显式模型或平台默认能力路由。
5. 由供应商网关发起请求并记录任务、日志和结果文件。

## 6. 数据对象设计

### 6.1 用户与教学对象

- `RequestUserContext`：承载用户 ID、姓名、角色、组织、课程上下文与认证状态。
- `TeachingCourse`：承载课程基础信息、负责人和组织归属。
- `TeachingAssignment`：承载课程作业与状态。
- `AssignmentSubmission`：承载学生提交、评分、评语与项目引用。

### 6.2 项目主对象

- `ScriptProject`：项目基础信息、拥有者、课程关联、风格参数、显式模型、审核状态、时间戳。
- `ScriptProjectAggregate`：项目聚合根，统一汇总项目下所有文档、文件、任务与审核记录。
- `ScriptProjectSummary`：列表页摘要对象，用于项目信息快速展示。

### 6.3 文档与版本对象

- `ScriptDocumentVersion`：原始剧本、完善稿、JSON 结构稿等文件版本元数据。
- `ScriptRevision`：修订快照，用于保存变更历史和恢复操作。
- `StoredFileRecord`：统一文件记录，支撑预览、下载、资源中心展示。

### 6.4 资产与镜头对象

- `ExtractedAsset`：人物、背景、道具等资产及其提示词、版本、扩展元数据。
- `KeyframeRecord`：关键帧图像、提示词、状态与模型调用信息。
- `StoryboardShot`：镜头拆分结果、脚本文本、运镜与资产引用。

### 6.5 流水线任务对象

- `VideoSegmentTask`：视频分段任务、状态、模型、错误信息和结果文件。
- `DubbingTask`：配音任务、输入文本、语言、音色与结果音频。
- `LipSyncTask`：口型同步任务，关联视频和音频输入。
- `VideoEditRenderTask`：剪辑渲染任务，区分预览与发布。
- `FinalCompositionTask`：最终成片任务。
- `ExportPackageTask`：导出包任务。
- `PipelineRun`：阶段运行记录，用于汇总运行中、成功数、失败数和当前阶段。

### 6.6 状态对象

项目已经使用多组状态枚举描述阶段推进，如：

- `ProjectStatus`：从 `DRAFT` 到 `COMPLETED`，覆盖脚本、资产、视频、配音、成片、导出等阶段。
- 各任务状态枚举：描述 `PENDING`、`QUEUED`、`RUNNING`、`SUCCESS`、`FAILED` 等执行结果。
- 审核状态枚举：描述提交、通过、驳回等审核结果。

状态对象的价值在于统一前后端显示与后端编排逻辑。

## 7. 模块协作关系

### 7.1 前端到后端

- 页面层负责收集用户输入和展示状态。
- Store 或 API 层统一封装请求。
- 后端控制器将接口按业务域暴露。
- 服务层处理业务规则、鉴权、编排和模型调用。
- 持久化层保存聚合对象、任务对象和文件索引。

### 7.2 项目域内部协作

项目域内部以 `ScriptProjectService` 和 `ScriptWorkflowService` 管理同步数据变更，以 `ScriptProductionOrchestrator` 管理异步长任务。

协作分工如下：

- `ScriptProjectService`：创建项目、导入剧本、保存脚本、恢复修订、查询聚合。
- `ScriptWorkflowService`：完善剧本、改写、续写、优化、抽取、镜头拆分等同步工作流步骤。
- `ScriptProductionOrchestrator`：视频、配音、口型同步、剪辑、导出等异步编排。
- `ContentReviewService`：审核状态流转与审批动作。
- `VideoEditingService`：剪辑草稿与渲染任务管理。

### 7.3 治理域协作

- 运营看板读取课程、提交、项目和审核统计数据。
- 媒体资源中心读取统一文件记录。
- 审计模块记录后台操作。
- 组织用户模块维护账号和组织归属。

治理域不直接修改项目内容，但影响项目访问、安全和平台可运营性。

## 8. 非功能设计约束

### 8.1 权限控制

- 所有课程、项目、审核与治理接口均依赖用户上下文。
- 项目读写能力需区分拥有者、教师评审者和管理员。
- 教学管理和治理后台入口需按角色裁剪前端导航与后端接口权限。

### 8.2 数据持久化

- 项目采用聚合保存思路，既持有数据库实体，也关联本地文件记录。
- 文件必须支持预览与下载，便于教学查看和媒体资源沉淀。
- 修订、任务、审核、导出等信息必须可追溯，避免只存最终结果。

### 8.3 并发与幂等

- 异步任务要限制重复提交，运行中任务不能被无意义叠加。
- 单个任务需要具备重试次数控制，避免无限失败重入。
- 视频等长任务采用线程池执行，并通过 `PipelineRun` 汇总状态。

### 8.4 容错与可观测性

- AI 返回失败、JSON 结构漂移、外部接口超时都要有明确错误信息。
- 路由日志、审计日志、运营看板共同构成平台可观测性基础。
- 局部失败应尽量局部恢复，而不是让整个项目重新开始。

## 9. 扩展方向

从现有聚合对象、页面路由和状态枚举判断，平台后续可以继续沿以下方向演进：

- 多人协同编辑与课程小组协作。
- 更完整的审核工作流与分级审批。
- 云端对象存储、外部交付平台对接与归档策略。
- 更细粒度的时间线编辑、素材编排和非编能力。
- 实时事件推送替代轮询，优化长任务体验。
- 更丰富的提示词模板市场、风格模板和项目预设。
- 更精细的资源计量、配额与成本分析能力。

## 10. 设计结论

当前项目已经形成清晰的项目化架构骨架：以角色与课程组织访问边界，以剧本工程承载生产主链路，以模型路由承载多供应商接入，以审核和导出承载交付闭环。后续设计应继续围绕以下主线演进：

- 保持前端入口按工作目标分区，而不是按技术模块分区。
- 保持后端以课程域、项目域、治理域、路由域解耦。
- 保持项目聚合能够承接更多阶段任务，而不破坏现有数据模型。
- 保持模型调用通过能力路由统一决策，减少业务代码耦合。
