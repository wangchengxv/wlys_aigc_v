# 项目需求分析

## 1. 文档目标

本文档基于当前仓库中的 `aigc-server`、`aigc-site-react`、根目录 `README.md` 与 `docs/script-project` 现有设计文档整理，描述平台当前已经形成的业务目标、角色诉求、核心场景、能力范围与非功能约束，用于统一项目级认知，并为后续概要设计、实施排期和验收提供依据。

## 2. 项目定位

本项目定位为一个面向校园教学与创作实训场景的 AIGC 平台，将课程组织、项目生产、内容审核、媒体资源管理与模型路由能力收敛到同一套系统中。平台不是单一的文生图或文生视频工具，而是围绕“教学发起 -> 项目创作 -> 审核交付 -> 资产沉淀”的完整链路展开。

从现有前后端实现来看，平台已经具备以下业务板块：

- 创作工具区：统一工作台、文生图、文生视频、图生视频、无限画布、三视图工具、历史记录。
- 课程与实训区：课程中心、作业管理、学生提交、教师评分与评审。
- 项目生产区：剧本工程、资产抽取、关键帧生成、视频生成、配音、口型同步、剪辑成片、导出交付。
- 治理与运营区：组织用户、审计日志、运营看板、媒体资源中心、设置中心。
- 模型与服务区：模型配置、服务商管理、统一路由、OpenAI/Anthropic 兼容代理。

## 3. 角色诉求

### 3.1 管理员

管理员关注平台的统一治理与可用性，核心诉求包括：

- 维护组织、班级、账号与权限边界，保证多角色访问安全。
- 管理模型、服务商、路由策略和全局配置，避免业务流程依赖单一厂商。
- 查看审计日志、媒体资源与运营看板，及时发现高频问题和资源沉淀情况。
- 为教学与项目生产提供稳定的基础设施，而不是直接参与单个项目的创作细节。

### 3.2 教师

教师是教学任务的发起者和项目评审者，核心诉求包括：

- 创建课程、布置作业、跟踪提交进度并完成评分或审核。
- 查看学生项目的剧本、资产、视频和导出结果，进行教学点评与过程指导。
- 在统一工作区中快速查看项目状态，减少在课程、项目、资源页之间频繁切换。
- 在保证教学节奏的同时，复用平台提供的模型、模板和项目工作流能力。

### 3.3 学生

学生是项目生产的直接执行者，核心诉求包括：

- 从课程任务快速进入个人项目，不重复搭建复杂创作环境。
- 基于剧本逐步完成资产抽取、关键帧、视频、配音、成片和导出。
- 能够回看历史结果、修改提示词与局部重试，不因单步失败丢失整个项目进度。
- 在权限允许范围内提交成果、接收评审意见并继续迭代。

### 3.4 访客/轻量入口用户

从首页与登录流程看，平台也支持未登录情况下的轻量入口模式，其诉求是：

- 快速理解平台分区和可用能力。
- 通过统一入口进入登录、创作或课程模块。
- 在最少学习成本下完成试用或演示。

## 4. 核心场景

### 4.1 教学组织场景

- 教师创建课程并发布作业。
- 学生进入课程后提交项目或更新项目内容。
- 教师查看提交记录、评分与评审意见，形成教学闭环。

### 4.2 AIGC 快速创作场景

- 用户在统一工作台中发起文本、图片、视频生成任务。
- 用户查看历史记录、下载结果、重生成内容或切换模型。
- 管理员或教师通过三视图工具、无限画布等增强工具辅助创作。

### 4.3 剧本驱动项目生产场景

- 用户创建剧本工程，支持文本录入和文件上传。
- 系统调用文本模型完善剧本，并生成结构化脚本。
- 用户在预览页进行改写、续写、三阶段优化，形成更适合视频生产的脚本素材。
- 系统抽取人物、背景、道具等资产，生成关键帧和视觉提示词。
- 系统拆分镜头并并发生成视频，随后继续配音、口型同步、剪辑成片与导出。

### 4.4 审核与交付场景

- 学生或项目拥有者提交内容审核。
- 教师或管理员审批、驳回并填写意见。
- 审核通过后启动导出包生成，沉淀交付文件。

### 4.5 平台治理与资源沉淀场景

- 管理员查看媒体资源中心，统一回收和追踪存储文件。
- 平台通过运营看板查看课程、提交、项目和审核状态。
- 审计日志记录关键动作，满足后台治理要求。

## 5. 能力范围

### 5.1 当前已覆盖的核心能力

结合现有前后端路由、控制器与实体设计，项目当前能力范围包括：

- 账号与角色模式：管理员、教师、学生三类角色，按功能区动态展示导航与访问入口。
- 课程管理：课程列表、课程详情、作业创建、作业状态流转。
- 提交与评审：学生提交项目，教师查看提交并评分、记录评审。
- 基础生成：文本、图像、视频生成，支持历史记录、模型选择和参数设置。
- 剧本工程：项目创建、上传导入、剧本完善、续写、改写、修订恢复、模型覆盖与提示词覆盖。
- 资产生产：人物/背景/道具抽取，视觉提示词、三视图、九宫格、关键帧生成与回退。
- 视频流水线：镜头拆分、视频分段生成、失败重试、状态轮询。
- 音视频后处理：配音、口型同步、剪辑预览、剪辑发布、成片生成、导出包生成。
- 审核交付：内容审核提交、审批、驳回与状态查询。
- 配置与治理：模型配置、服务商目录、路由控制台、媒体资源中心、审计日志、运营看板。

### 5.2 当前能力边界

当前版本更强调“单校区/单平台内的一体化工作流”，主要边界包括：

- 数据持久化以本地文件与数据库混合方式为主，强调单机可恢复与后台管理便利。
- 剧本项目已支持多阶段生产链路，但不等于成熟的专业非编系统。
- 模型调用以统一路由和能力映射为核心，允许项目级显式覆盖，但仍以平台内的服务商配置为前提。
- 教学场景与项目生产已打通，但多人协同、实时协作、复杂审批流等能力仍属于扩展方向。

### 5.3 本轮文档重点关注的业务闭环

项目级分析重点关注以下闭环，因为它们在代码中已经形成清晰的对象与接口体系：

1. 课程 -> 作业 -> 提交 -> 评审闭环。
2. 项目 -> 剧本 -> 资产 -> 视频 -> 配音/口型同步 -> 成片 -> 导出闭环。
3. 模型配置 -> 路由 -> 平台调用 -> 日志统计闭环。
4. 审核 -> 发布 -> 资源沉淀 -> 运营看板闭环。

## 6. 非功能约束

### 6.1 权限与安全

- 平台必须基于角色和上下文做访问控制，最少区分管理员、教师、学生。
- 项目读取、写入、评审和课程查看都要与拥有者、课程、组织关系联动。
- 审计日志与后台治理能力需要保留关键操作记录，支持问题追溯。

### 6.2 稳定性与可恢复性

- 剧本工程与生成流水线属于长链路任务，系统必须支持任务状态恢复和结果追踪。
- 单个分段任务失败时不能拖垮整个项目，应支持局部重试。
- 服务重启后应尽可能从持久化数据中恢复项目、资产和流水线状态。

### 6.3 性能与并发

- 视频、配音、口型同步、剪辑等耗时操作必须异步执行，避免阻塞请求线程。
- 并发任务需要受配置约束，避免触发外部模型限流。
- 列表页、看板和媒体资源页需要可控分页与过滤能力，避免后台访问过重。

### 6.4 可扩展性

- 模型能力要按文本、图像、视频、TTS 等能力进行路由，不与固定厂商耦合。
- 项目聚合对象要能自然承接新增阶段，如审核、导出、成片或更多 AI 工具链。
- 提示词模板与项目级模型覆盖应允许后续扩展，而不是写死在单个流程中。

### 6.5 可维护性

- 前端模块应按“概览、创作、教学、项目、治理、资源”分区组织，降低认知负担。
- 后端接口应围绕课程域、项目域、路由域、资源域拆分，减少超大控制器以外的业务耦合。
- 关键状态枚举、任务对象、文件对象需要保持命名稳定，便于数据库、接口与前端联动。

### 6.6 数据一致性

- AI 输出的结构化 JSON 需要校验和兜底，避免后续资产抽取、镜头拆分失败。
- 文件记录、任务记录与项目聚合之间必须可追踪，避免出现资源文件无法归属的问题。
- 项目状态要和流水线阶段同步变化，不能出现前端看到的阶段与后端真实状态长期不一致。

## 7. 需求分析结论

从现有实现可以看出，本项目的核心价值不是提供零散 AI 能力，而是在校园实训场景下把“角色、课程、项目、审核、资源、模型”组织成统一平台。需求上的关键不在单点功能是否足够多，而在于以下几点是否持续成立：

- 教师和学生能否在课程上下文中完成项目协作与提交。
- 项目生产是否围绕剧本工程形成可逐步推进、可回退、可审核的流水线。
- 平台是否能在不绑定单一厂商的前提下调用不同模型能力。
- 管理员是否能通过治理、资源和日志能力维持平台稳定运行。

因此，后续设计与开发应继续坚持“项目化生产、教学化管理、平台化治理、路由化模型接入”的总体方向。
