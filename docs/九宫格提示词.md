# 九宫格分镜提示词系统 · 完整版 v2.0

> 覆盖从镜头拆分 → 结构校验 → 图像生成 → 中文展示 → 批量改写 → 视频成片的完整工作流。
> 每条提示词均附使用说明与优化注记。

---

## 执行链路总览

```
① splitSystem + splitUser
        ↓ panels JSON
② JSON 校验（数量 · 索引 · 多样性）
        ↓ 校验通过
③ imagePrefix + imagePanelTemplate × N + imageSuffix + imageNoTextConstraint
        ↓ 网格分镜图
④ translatePrompt / rewritePrompt（按需）
        ↓
⑤ sora2NineGrid + nineGridGuardrails → 视频成片
```

> **错误前置原则**：结构拆分阶段的 JSON 质量直接决定下游成功率。建议先稳定 splitUser 输出，再推进后续环节。

---

## 01 结构拆分提示词

将单一镜头动作拆解为 N 个不重复视角的结构化 JSON。这是整个工作流最关键的环节——输出的 panels 数组质量直接决定下游图像与视频的效果。

### 1.1 splitSystem — 角色设定与硬边界

**角色**：`system`

```
你是一位资深电影分镜师，精通多机位叙事与镜头语言节奏设计。

你的核心任务是：将同一个镜头动作拆解成 {panelCount} 个视角互不重复的分镜面板，用于生成 {gridLayout} 九宫格分镜图。

【角色定位】
- 你以"导演的眼睛"审视每一格，而不是机械罗列视角
- 每个面板必须是可独立传达信息的电影关键帧
- 整个格网必须作为一个整体叙述一个完整的动作或情绪弧线

【硬边界约束】
1. 网格布局：严格执行 {layoutInstruction}，不得增删行列
2. 面板数量：恰好 {panelCount} 个，index 从 0 到 {lastIndex}，不得重复
3. 景别多样性：{panelCount} >= 6 时，shotSize 至少涵盖 3 种（如远景/中景/特写）
4. 视角去重：任意两格的 [shotSize + cameraAngle] 组合不得完全相同
5. 角色连续性：跨格保持角色外观、服装、主运动方向一致
6. 场景连续性：光线、色调、道具位置保持前后一致

【叙事节奏规则】
- index=0：建立场景全貌（orienting shot），交代环境与人物关系
- index=1 至 {midIndex}：逐步推进动作，切换视角揭示细节或情绪
- index={lastIndex}：情绪落点或动作结果，给观众明确的叙事收尾

【输出格式】
只输出合法 JSON，不附带任何说明文字。
格式：{"panels":[{"index":0,"shotSize":"","cameraAngle":"","description":""},...]}
```

**优化注记**
- 新增"导演的眼睛"角色定位，避免模型机械罗列视角，强化叙事感知
- 明确 midIndex 概念，让模型理解节奏推进的中段逻辑
- 角色与场景连续性拆分为两条独立规则，减少歧义
- 输出格式示例直接内嵌，降低模型格式混淆概率

---

### 1.2 splitUser — 完整拆分指令

**角色**：`user`

```
请将以下镜头动作拆解为 {panelCount} 个不同的摄影视角，生成一张 {gridLayout} 网格分镜图所需的结构化面板数据。

━━━ 网格硬约束 ━━━
布局规格：必须严格为 {layoutInstruction}
读取顺序：从左到右、从上到下
行列示意：{layoutExample}
补充限制：{layoutSpecificConstraint}

━━━ 镜头上下文 ━━━
【动作摘要】    {actionSummary}
【镜头运动】    {cameraMovement}
【场景地点】    {location}
【时间氛围】    {time} · {atmosphere}
【在场角色】    {characters}
【视觉风格】    {visualStyle}

━━━ 输出规则（严格执行，违反则重新生成）━━━
R1  顶层键名固定为 "panels"，值为数组，不附带其他键
R2  数组长度恰好为 {panelCount}；每项含显式 index 字段，值域 0–{lastIndex}，不重复
R3  每项必须包含非空字段：shotSize / cameraAngle / description
R4  shotSize 与 cameraAngle 使用简短中文（如：大全景 / 俯角 45°）
R5  description 为英文单句，10–30 词，描述画面内容与构图重点
R6  视角多样性：[shotSize + cameraAngle] 组合全局唯一；{panelCount}≥6 时至少含 3 种 shotSize
R7  叙事节奏：index=0 建立场景 → 中间格逐步推进 → 最后一格呈现结果或情绪落点
R8  连续性：角色外观/服装/道具/主运动方向跨格一致；光影色调跨格一致
R9  禁止在 description 中出现可读文字、数字标注、UI 元素描述
R10 只输出 JSON，不输出任何解释、注释或 Markdown 代码块标记
```

**优化注记**
- 采用"━━━"分区结构替代扁平格式，大幅提升模型对上下文分区的识别准确率
- R1–R10 编号规则体系，便于调试时定位具体违规项
- R9 新增（禁止 description 含文字描述），与图像生成阶段的无文字约束联动
- R10 强调禁止 Markdown 代码块标记，防止 ```json 包裹导致解析失败

## 02 图像生成提示词

采用"前缀 + 逐格中段 + 后缀 + 最高优先级无文字约束"四段式拼接。每段职责单一，可独立维护与替换。

**拼接顺序**：imagePrefix → imagePanelTemplate（循环 {panelCount} 次）→ imageSuffix → imageNoTextConstraint（始终置于最末）

---

### 2.1 imagePrefix — 整体网格定义

```
Create ONE single cinematic storyboard contact sheet image.

GRID SPECIFICATION (non-negotiable):
- Layout: exactly {layoutInstruction} ({panelCount} equal panels)
- Separators: thin white lines (2–3px), uniform between all panels
- Panel order: {layoutExample} (left-to-right, top-to-bottom)
- {layoutSpecificConstraint}

VISUAL STYLE: {visualStyle}
SCENE CONTEXT: {sceneContext}

CONSISTENCY RULES across all panels:
- Same character appearance, costume, props, and dominant movement direction
- Same lighting direction, color temperature, and mood
- Same background depth and environmental details
- No panel may be blank, blurred, or decorative only

COMPOSITION:
- Every panel must be a complete, readable cinematic keyframe
- Vary shot size and camera angle between panels to build visual rhythm
- All {panelCount} panels must be present, identical in size, and fully rendered

Panels (left-to-right, top-to-bottom):
```

**优化注记**
- 新增 `{sceneContext}` 参数，让图像模型对场景有整体感知
- 分隔线规格细化为 2–3px 白线，减少图像模型自行发挥的空间
- CONSISTENCY RULES 独立分区，强调跨格一致性是全局约束

---

### 2.2 imagePanelTemplate — 单格描述模板

```
Panel {displayIndex} | {position} | [{shotSize} / {cameraAngle}]
{description}

```

> `{displayIndex}` 使用 1-based（Panel 1–9），`{position}` 使用"Row 1 Col 1"格式，双重定位减少错位概率。末尾空行作为面板间分隔符。

---

### 2.3 imageSuffix — 布局与风格收口约束

```
━━━ HARD LAYOUT CONSTRAINTS ━━━
- Output EXACTLY ONE storyboard grid image (no variants, no alternatives)
- Grid must be {layoutInstruction}, exactly {panelCount} panels total
- All panel sizes must be identical — no merged cells, no oversized panels, no inset panels
- Do NOT add extra rows, extra columns, blank panels, missing panels, or alternate layouts
- {layoutSpecificConstraint}

━━━ STYLE CONSTRAINTS ━━━
- Cinematic color grading consistent with: {visualStyle}
- No vignette effects that obscure panel edges
- Separator lines must remain visible and straight
- No decorative borders, watermarks, or frame numbers on panels

━━━ CHARACTER & SCENE INTEGRITY ━━━
- Character face, body proportions, and costume must match across all panels
- Background environment must be consistent (same location, same time of day)
- Lighting angle and color temperature must not shift between panels
- Action continuity: subject's motion should feel sequential across panels
```

**优化注记**
- 拆分为三个独立约束区块（布局/风格/角色完整性），优先级层次清晰
- 新增"分隔线可见且笔直"约束，防止模型生成模糊或弯曲的网格线
- 新增"禁止在面板上叠加帧号"，防止图像模型自动添加数字标注
- 新增 ACTION CONTINUITY 概念，引导模型理解跨格的动作连续性

---

### 2.4 imageNoTextConstraint — 无文字硬约束 ⚠️ 最高优先级

```
━━━ ABSOLUTE NO-TEXT RULE (HIGHEST PRIORITY — overrides all other instructions) ━━━

This storyboard grid image must contain ZERO readable text in every single panel.

FORBIDDEN text elements (apply to every panel without exception):
✗ Letters, words, sentences in any language
✗ Arabic numerals or any numeral system
✗ Subtitles, captions, dialogue bubbles, thought bubbles
✗ Logos, brand marks, watermarks, copyright notices
✗ Signage, street signs, storefront text, building labels
✗ UI labels, HUD elements, screen text, notification banners
✗ Handwritten notes, graffiti, tattoo text, printed labels

SPECIAL RENDERING RULES for text-containing objects:
→ Signs / storefronts: render as blank boards or indistinct color patches
→ Books / documents / newspapers: render pages as blank or abstract texture
→ Screens / monitors: show abstract light glow or blurred imagery only
→ Clothing text / badges: replace with solid color or abstract pattern
→ Whiteboards / chalkboards: render as blank surface

This rule cannot be overridden by scene descriptions, character details, or any other instruction.
```

**优化注记**
- 从约束列表升级为完整独立模块，强化"最高优先级且不可被覆盖"的语义
- 新增"特殊对象渲染规则"，明确告知模型如何处理书本/屏幕/招牌等容易触发文字生成的场景元素
- ✗ 符号强化禁止语义，视觉权重比纯文字列表更高
- 末行补充"此规则不可被任何其他指令覆盖"，防止多指令冲突时妥协

## 03 面板描述翻译提示词

将英文 description 转为中文界面展示文本。职责单一：忠实翻译，不添加剧情，不改变构图意图。

### translatePrompt — 面板批量翻译

**角色**：`user`

```
你是影视分镜翻译编辑，专注于将英文分镜描述转化为简洁、有画面感的中文界面展示文本。

【翻译原则】
1. 忠实原句的主体、动作、构图重点与镜头意图，不增减剧情信息
2. 保留专业摄影词汇的中文对应（如 shallow depth of field → 浅景深）
3. 风格上追求"导演手记"语感：精准、克制、有画面张力
4. 禁止添加情绪修饰词或主观评价（如"震撼的""美丽的"）

【输入 JSON】
{panelsJson}

【输出规则（严格执行）】
O1  顶层结构：{"translations":[...]}
O2  translations 数量恰好为 {expectedCount}，index 完整覆盖 0–{expectedCountMinusOne}，不重复
O3  每项格式：{"index": number, "descriptionZh": "string"}
O4  descriptionZh 建议 15–40 字，句式简洁（主语+动作+构图/氛围）
O5  保留英文原文中的镜头术语对应中文（不音译，选用业界通用译法）
O6  只输出 JSON，不输出解释、注释或 Markdown 代码块标记
```

**优化注记**
- 新增翻译原则区块，明确"导演手记"语感定位，避免口语化或过于文学化的翻译
- 增加"禁止情绪修饰词"规则，保持分镜描述的客观性与专业性
- O5 新增摄影术语保留规则，防止专业词汇错译

## 04 面板批量改写提示词

根据用户自然语言指令对现有 panels 进行风格、镜头或情绪方向的调整，同时强制保持结构完整性与叙事连贯性。

### rewritePrompt — 面板批量改写

**角色**：`user`

```
你是电影分镜提示词编辑器，专注于在保持结构与叙事连贯性的前提下，根据导演指令对分镜面板进行精准改写。

━━━ 改写指令 ━━━
{instruction}

━━━ 镜头上下文（改写需与此保持一致）━━━
动作摘要：{actionSummary}
镜头运动：{cameraMovement}
场景：{sceneLocation} · {sceneTime} · {sceneAtmosphere}
视觉风格：{visualStyle}

━━━ 当前面板数据 ━━━
{panelsJson}

━━━ 改写规则 ━━━
R1  仅修改用户指令涉及的描述维度，未被指令覆盖的内容保持原文
R2  输出的 panels 数量恰好为 {expectedCount}，index 完整覆盖 0–{expectedCountMinusOne}
R3  每项包含非空的 shotSize / cameraAngle / description
R4  shotSize 与 cameraAngle 使用简短中文；description 为英文单句，10–30 词
R5  [shotSize + cameraAngle] 组合全局唯一；shotSize 至少包含 {requiredShotSizeKinds} 种
R6  叙事节奏保持：index=0 建立场景，最后一格情绪落点，中间格逐步推进
R7  角色外观/服装/道具/运动方向跨格一致；光影色调跨格一致
R8  description 中禁止出现可读文字、数字标注、UI 元素描述
R9  只输出 JSON（顶层键 "panels"），不输出解释或 Markdown 代码块标记

━━━ 改写质量自检（输出前执行）━━━
□ 指令要求是否已全部体现？
□ 是否存在 [shotSize + cameraAngle] 重复组合？
□ 叙事节奏是否完整（建立→推进→落点）？
□ 角色与场景连续性是否跨格一致？
```

**优化注记**
- 新增 R1"最小改写原则"：只修改被指令覆盖的维度，避免改写时模型"顺手"改动无关内容
- 新增"改写质量自检"区块（□ checklist），引导模型输出前进行内部验证，显著降低格式与连贯性错误率
- R8 与 splitUser 的 R9 保持联动，确保改写后的 description 同样不含文字描述

## 05 视频生成提示词

以九宫格图作为镜头顺序参考，生成单画面全屏顺序切镜视频。
**核心原则**：网格图永远是"顺序表"而不是"输出内容"。

> ⚠️ **最高风险**：模型将参考图的网格布局直接渲染为视频内容（分屏/拼贴）。guardrails 提示词必须作为附加参数拼接，不可省略。

---

### 5.1 sora2NineGrid · 中文版 — 主体视频指令

```
⚠️ 参考图说明（最高优先级）：
附件为 {gridLayout} 网格分镜板（共 {panelCount} 格），该图像仅用于提供镜头顺序参考。
网格图本身绝对不得作为视频的可见内容、背景或任何画面元素出现。

━━━ 视频核心要求 ━━━
第一帧：必须是面板 1 的全屏场景（单画面占满 100% 屏幕，无任何网格线或拼贴痕迹）
动作描述：{actionSummary}
镜头运动：{cameraMovement}
视觉风格：{visualStyle}

━━━ 分镜顺序（按参考图从左到右、从上到下）━━━
{panelDescriptions}

━━━ 切镜规则 ━━━
- 镜头按 1 → {panelCount} 顺序逐个出现，禁止跳序或同时展示多个面板
- 任意时刻只允许一个全屏镜头，禁止多画面同时运动
- 每个视角约 {secondsPerPanel} 秒
- 转场方式：硬切或自然动机转场（不用淡入淡出或交叉溶解，保持电影节奏）
- 跨镜头保持角色外观一致、动作连续

━━━ 音频规则 ━━━
- 允许 {language} 配音或旁白
- 禁止字幕、任何画面叠字、片头片尾文字
```

**优化注记**
- 参考图说明提升为独立首段，优先建立"网格仅为参考"认知
- 明确转场偏好（禁用淡入淡出），避免模型默认使用缓慢转场破坏电影节奏感
- 新增"跨镜头动作连续性"要求

---

### 5.2 nineGridGuardrails · 中文版 — 硬约束兜底 ⚠️ 最高优先级

```
━━━ 硬性规则（最高优先级，不可被任何其他指令覆盖）━━━

RULE-1  视频自始至终必须是单画面全屏输出：
        任意时刻仅有一个镜头占满 100% 画面，不得存在任何形式的画面分割。

RULE-2  严格禁止以下画面形式（违反则视为生成失败）：
        ✗ 九宫格 / 六宫格 / 四宫格等任何分屏布局
        ✗ 拼贴 / 缩略图墙 / 画中画 / 多窗口并行播放
        ✗ 将参考网格图的任意部分作为视频可见内容

RULE-3  镜头切换顺序：
        必须严格按 1 → {panelCount} 逐格切换，禁止：
        ✗ 多个面板同时出现或同时运动
        ✗ 跳序切换（如 1→3→2）
        ✗ 回切（如 3→1）

RULE-4  冲突解决优先级：
        "单画面全屏 + 1→{panelCount} 顺序切镜" > 还原网格参考图外观
        当上述两个目标冲突时，必须牺牲网格外观还原，保证单画面全屏与顺序切镜。
```

**优化注记**
- RULE-1 至 RULE-4 编号化，调试时可精确引用具体规则号
- 新增 RULE-3 的禁止跳序与回切规则，填补原版仅约束"同时出现"的盲区
- RULE-4 明确冲突解决优先级顺序，给模型提供明确的决策树

---

⚠️ REFERENCE IMAGE NOTICE (HIGHEST PRIORITY):

━━━ VIDEO REQUIREMENTS ━━━

━━━ AUDIO RULES ━━━
- Voiceover / narration in {language} is permitted

```

### 5.4 nineGridGuardrails · 英文版 — 硬约束兜底 ⚠️ 最高优先级

```
━━━ HARD RULES (HIGHEST PRIORITY — cannot be overridden by any other instruction) ━━━

RULE-1  The video must be single-shot full-screen at all times:
        Exactly one shot must occupy 100% of the frame at every moment; no frame splitting permitted.

RULE-2  The following formats are strictly forbidden (violation = generation failure):
        ✗ Any split-screen layout (3×3, 2×3, 2×2, or any grid)
        ✗ Collage, thumbnail wall, picture-in-picture, multi-window playback
        ✗ Any portion of the reference grid image appearing as visible video content

RULE-3  Shot transition order:
        Must transition strictly 1 → {panelCount}, one shot at a time. Forbidden:
        ✗ Simultaneous display or animation of multiple panels
        ✗ Out-of-order transitions (e.g., 1→3→2)
        ✗ Reverse cuts (e.g., 3→1)

RULE-4  Conflict resolution priority:
        "Single full-screen shot + sequential 1→{panelCount} cuts" > restoring grid reference appearance.
        When these goals conflict, sacrifice grid appearance fidelity; preserve single full-screen sequential cuts.
```

## 07 JSON 校验逻辑参考（伪代码）

在 splitUser 输出通过以下校验后，才进入图像生成阶段：

```javascript
function validatePanels(result, expectedCount) {
  // 1. 顶层结构检查
  if (!result.panels || !Array.isArray(result.panels))
    throw "缺少 panels 数组";

  // 2. 数量检查
  if (result.panels.length !== expectedCount)
    throw `面板数量错误：期望 ${expectedCount}，实际 ${result.panels.length}`;

  // 3. 索引完整性检查
  const indices = result.panels.map(p => p.index).sort((a, b) => a - b);
  const expected = Array.from({ length: expectedCount }, (_, i) => i);
  if (JSON.stringify(indices) !== JSON.stringify(expected))
    throw `索引不连续或有重复：${indices}`;

  // 4. 必填字段检查
  for (const panel of result.panels) {
    if (!panel.shotSize || !panel.cameraAngle || !panel.description)
      throw `面板 ${panel.index} 存在空字段`;
    if (panel.description.split(' ').length < 8)
      throw `面板 ${panel.index} description 过短（< 8词）`;
  }

  // 5. 视角多样性检查
  const combos = result.panels.map(p => `${p.shotSize}|${p.cameraAngle}`);
  if (new Set(combos).size !== combos.length)
    throw `存在重复的 [shotSize + cameraAngle] 组合`;

  // 6. 景别多样性检查（当 panelCount >= 6）
  if (expectedCount >= 6) {
    const shotSizes = new Set(result.panels.map(p => p.shotSize));
    if (shotSizes.size < 3)
      throw `景别种类不足 3 种，当前：${[...shotSizes].join(', ')}`;
  }

  return true; // 校验通过，可进入图像生成
}
```

## 08 落地顺序建议

1. **先稳定 splitUser 的 JSON 产出质量**，用校验逻辑前置拦截格式错误
2. **再调 imagePrefix/Panel/Suffix 的布局一致性**，重点验证网格线与面板尺寸
3. **将 imageNoTextConstraint 设为默认常驻**，任何图像请求都拼接此约束
4. **对 sora2NineGrid + guardrails 进行成片验证**，重点检查是否出现分屏/拼贴问题

> 把错误尽量前置在"结构化输出"阶段，可显著降低图像和视频阶段的返工成本。
