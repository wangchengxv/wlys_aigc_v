# 视频下载与预览修复实施计划

## 目标

* 解决“视频模型服务返回异常，缺少可用视频地址”导致前端无法播放/下载的问题。

* 保证视频生成成功后，前端可以稳定预览与播放。

* 实现“直接调用浏览器下载能力”的下载路径，提升可用性与兼容性。

## 已定位事实

* 报错由后端视频URL解析逻辑抛出：在轮询成功分支中，若未解析到URL即抛异常。

* 前端当前已有基础播放与下载入口，但下载方式仅是简单a标签触发，缺少更稳健的兜底与可视化预览交互。

## 实施步骤

### 1. 后端：增强视频URL解析与轮询兼容性

1. 梳理视频模型返回结构中的多种URL字段来源（顶层、data内、results数组、可能的嵌套对象）并统一提取策略。
2. 重构/增强URL提取函数：支持更多候选字段与数组路径，保持去重、非空过滤与http(s)合法性检查。
3. 调整轮询成功态逻辑：当状态成功但URL暂时为空时，按现有轮询策略进行有限重试，不立即抛“缺少可用视频地址”。
4. 在达到最大重试后仍无URL时，返回更可诊断的错误信息（包含任务状态与关键响应片段摘要，避免泄露敏感信息）。
5. 保持对图片流程无副作用，确保只影响视频分支。

### 2. 后端：输出前端可消费的标准视频结果

1. 确认返回给前端的`videoResults`总是数组结构，元素为可直接访问的URL字符串。
2. 若第三方仅返回单个URL，统一包装为数组；若无URL则走可追踪错误分支而非静默成功。
3. 补充必要的空值保护，避免NPE或空集合误判。

### 3. 前端：实现浏览器原生下载优先策略

1. 封装视频下载方法：优先使用`fetch -> blob -> URL.createObjectURL -> a[download]`触发浏览器下载。
2. 为跨域或鉴权受限场景增加兜底：若blob下载失败，自动回退到直接打开URL并提示用户使用浏览器保存。
3. 统一下载文件名策略（时间戳/序号），并确保下载后释放临时Object URL。
4. 在UI中反馈下载状态（进行中/成功/失败）与明确提示文案。

### 4. 前端：开发更完整的视频预览能力

1. 在现有卡片播放基础上新增“预览”交互（弹层/模态）用于大尺寸播放。
2. 预览层内提供标准视频控制能力（播放、进度、音量、全屏入口）并显示加载与错误状态。
3. 保留“新窗口打开”作为备选播放路径。
4. 对无效URL、加载失败、网络异常给出明确提示，避免空白区域。

### 5. 联调与验证

1. 构造至少三类场景验证：正常返回URL、延迟返回URL、成功态但最终无URL。
2. 验证前端链路：生成后自动展示视频、预览可播放、下载可触发并拿到文件。
3. 验证异常链路：失败提示准确，且不会阻塞后续再次生成。
4. 使用浏览器实际点击下载按钮验证“直接调用浏览器下载功能”的用户路径。

### 6. 回归与收尾

1. 回归图片生成与现有工作台交互，确保无回归问题。
2. 整理改动说明：后端解析兼容点、前端下载策略、预览功能入口与异常处理行为。

## 实施顺序

1. 先改后端（步骤1-2），确保稳定产出可用URL。
2. 再改前端下载与预览（步骤3-4）。
3. 最后联调验证与回归（步骤5-6）。

## 风险与对策

* 风险：第三方返回结构继续变化。\
  对策：提取逻辑做多路径兼容，并保留有限重试与可诊断错误。

* 风险：跨域策略影响blob下载。\
  对策：设计自动回退方案，不阻断用户通过浏览器保存视频。

* 风险：大文件预览卡顿。\
  对策：预览层增加加载状态与失败提示，避免界面假死。
