# AI Change Gate 模板（任务启动前必填）

## 1. 模板目标

用于约束 AI Agent 在任何代码改动前，先完成工程理解与风险评估，避免“只看单文件就直接改”的盲改行为。

## 2. 使用方式

1. 在接到需求后、开始读写代码前，先完整填写本模板。
2. 严格按“固定输出结构”输出，不得删减必填项。
3. 若触发“强制门禁条件”，必须先通过 Gate，才可进入实现。

## 3. 固定输出结构（必须原样保留标题）

> 说明：以下结构为固定骨架。执行时仅替换方括号占位符内容，不可删除章节。

### A. 任务摘要

- 需求原文：[粘贴用户需求]
- 目标结果：[一句话定义交付结果]
- 非目标范围：[明确本次不做什么]

### B. 上下文与证据

- 关联文件/模块：[路径1]、[路径2]、[路径3]
- 关键证据（代码/文档事实）：[证据1]；[证据2]
- 现状结论：[当前行为、约束、已知边界]

### C. 影响面评估

- 直接影响：[页面/接口/服务/状态]
- 间接影响：[共享组件、公共方法、配置、数据结构]
- 回归风险点：[风险1]、[风险2]

### D. 方案与改动边界

- 拟改动点：[文件 + 变更意图]
- 不改动点：[明确禁止触碰区域]
- 备选方案：[方案A/方案B + 取舍理由]

### E. 验证与回滚

- 最小验证路径：[静态检查/单测/手测路径]
- 通过标准：[可观测结果]
- 回滚策略：[如何快速撤回与止损]

### F. Gate 判定（必须给出结论）

- Gate 结果：`GO` / `NO-GO`
- 判定理由：[2-3条依据]
- 下一步动作：[进入实现 / 补充信息后重评]

## 4. 禁止盲改规则（硬性）

1. 禁止在未完成 A-F 全部章节时直接改代码。
2. 禁止只基于单文件片段下结论，至少要给出“上下游影响”说明。
3. 禁止未声明“非目标范围”就扩大改动。
4. 禁止未给出验证路径就提交实现。
5. 禁止遇到信息缺口时“猜测实现”，必须先补充证据或向用户澄清。
6. 禁止改动未纳入“拟改动点”的文件；若需扩展范围，先重跑 Gate。

## 5. 触发条件（任一命中即强制 Gate）

- 首次接手该仓库或首次接手该功能域。
- 涉及跨模块改动（前后端联动、共享状态、公共组件、公共工具）。
- 涉及数据结构/接口契约/配置项变更。
- 需求描述不完整、存在歧义或验收标准不清晰。
- 涉及高风险区域（鉴权、支付、上传下载、任务流水线、持久化、全局状态）。
- 需要修改 3 个及以上文件，或预计影响 2 个及以上页面/服务。

## 6. 可复用示例

### 示例 1：页面文案 + 交互小改（低风险）

```md
### A. 任务摘要
- 需求原文：将“立即生成”按钮文案改为“开始生成”，并补充 loading 态禁用
- 目标结果：按钮文案更新且重复点击被禁用
- 非目标范围：不改接口、不改后端

### B. 上下文与证据
- 关联文件/模块：src/pages/WorkspacePage.tsx；src/components/workspace/GeneratePanel.tsx
- 关键证据：按钮文案来自 GeneratePanel；loading 状态由页面状态透传
- 现状结论：仅前端局部改动即可满足

### C. 影响面评估
- 直接影响：工作台按钮展示与点击行为
- 间接影响：无共享契约变更
- 回归风险点：按钮禁用条件误判导致不可点击

### D. 方案与改动边界
- 拟改动点：GeneratePanel 按钮文案与 disabled 逻辑
- 不改动点：API 请求、后端生成逻辑
- 备选方案：A 在组件层处理；B 在页面层处理（选 A，影响更小）

### E. 验证与回滚
- 最小验证路径：手测点击 + loading 中重复点击拦截
- 通过标准：文案正确、loading 时不可重复触发
- 回滚策略：回退组件单文件改动

### F. Gate 判定（必须给出结论）
- Gate 结果：GO
- 判定理由：影响单模块、无契约变更、验证路径清晰
- 下一步动作：进入实现
```

### 示例 2：接口字段调整（中高风险）

```md
### A. 任务摘要
- 需求原文：`/api/v1/tasks/{id}` 返回新增 `durationMs`，前端列表展示耗时
- 目标结果：后端返回新字段，前端列表可展示
- 非目标范围：不重构任务模型历史字段

### B. 上下文与证据
- 关联文件/模块：后端 Task DTO/Controller；前端 api client；HistoryPage
- 关键证据：当前响应模型无 `durationMs`；前端列表字段映射固定
- 现状结论：需前后端联动并校验兼容性

### C. 影响面评估
- 直接影响：任务详情接口、历史列表渲染
- 间接影响：类型定义、序列化、潜在缓存字段
- 回归风险点：旧数据无字段导致前端渲染异常

### D. 方案与改动边界
- 拟改动点：后端响应 DTO + 映射逻辑；前端类型与渲染
- 不改动点：任务调度核心流程
- 备选方案：A 后端计算实时值；B 持久化字段（先选 A，最小入侵）

### E. 验证与回滚
- 最小验证路径：后端接口返回校验 + 前端页面展示校验
- 通过标准：新老任务均可正常渲染，缺失字段降级为 `--`
- 回滚策略：移除前端展示并保持后端字段向后兼容

### F. Gate 判定（必须给出结论）
- Gate 结果：GO
- 判定理由：已识别跨端影响、定义降级策略、验证口径明确
- 下一步动作：按拟改动点实施
```

## 7. 输出自检清单（提交前）

- 是否保留了 A-F 全部固定标题。
- 是否明确了“非目标范围”和“不改动点”。
- 是否给出了可执行的验证路径与通过标准。
- 是否在命中触发条件时执行了强制 Gate。
- 是否给出 `GO` 或 `NO-GO` 明确结论。
